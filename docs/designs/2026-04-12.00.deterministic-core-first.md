# Deterministic Core First

## Design Goal

Define the first implementation slice for Flexnet as a deterministic blockchain execution kernel, without real peer-to-peer networking or live consensus.

The objective of this slice is to make block execution, state transition, hashing, and validation fully deterministic and testable across many scenarios.

## Agreed Scope

- No real p2p networking in v1.
- No live validator discovery or node election in v1.
- No fork choice or distributed liveness work in v1.
- The first implementation target is a single-node deterministic engine that any number of simulated nodes can run and compare.

## Agreed Architecture

Use a single crate for the first implementation:

- `flexnet-core`

This crate owns:

- protocol data types
- canonical serialization used by hashing and signatures
- state representation
- transaction verification
- transaction application
- block verification
- block execution
- genesis loading and validation

Do not split this into multiple crates yet. The current scope does not justify extra package boundaries.

## Model

Flexnet is modeled as a deterministic state machine:

- the genesis state defines the initial state
- each block contains an ordered list of transactions
- executing transactions in order against the previous canonical state produces the next state
- the resulting state is committed by `state_hash`

The chain is therefore an ordered log of state transitions with cryptographic commitments, not a stored list of full states in every block.

## State Model

The first state model is account-based:

- `Address = [u8; 32]`
- `Account = { balance: u128, nonce: u128 }`
- `State = map<Address, Account>`

Implicit account rule:

- missing accounts behave as `{ balance: 0, nonce: 0 }`
- empty accounts are excluded from `state_hash`

## Transaction Model

The first transaction set contains only one operation:

- `Transfer`

`Transfer` remains the only supported transaction in the first implementation slice. Additional transaction kinds are intentionally deferred.

## Determinism Invariants

- all integer serialization is little-endian
- all hashes use sha256
- all signatures use ed25519
- state hashing uses byte-sorted addresses
- transaction execution order is the order they appear in the block
- if any transaction in a block is invalid, the entire block is invalid
- block validity is a pure function of prior canonical state and block bytes

## Testing Strategy

Testing should focus on deterministic scenario coverage rather than networking:

- valid transfer execution
- invalid chain id and version
- invalid nonce
- self-transfer rejection
- zero-amount rejection
- insufficient balance
- balance and nonce overflow
- invalid signature
- block rejection when any included transaction is invalid
- state hash stability
- block hash stability
- replaying the same block sequence always yields identical final state and block hashes

Multi-node behavior should be tested by running multiple independent executions over the same inputs and asserting byte-for-byte identical outputs.

## Crate Naming Decision

Use `flexnet-core` as the implementation crate for this phase.

Reasons:

- it matches the current scope exactly
- it keeps the repo structurally simple
- it avoids premature crate fragmentation
- it leaves room for later crates like `flexnet-sim` or `flexnet-cli` if they become necessary

## Open Issues

These should be resolved before or during implementation:

1. Future transaction extensibility:
   The current signature payload does not include a transaction kind tag because only `Transfer` exists. If additional transaction types are added later without a version bump, the signing format must change.
2. Genesis encoding:
   The spec describes genesis as JSON, but JSON object field order is not canonical. Implementations must parse then sort by address bytes before hashing.
3. State storage boundary:
   The implementation should distinguish between in-memory execution state and any later persistence format. Persistence is out of scope for this phase.

## Rejected Alternatives

- Implementing real networking first:
  rejected because it adds nondeterministic surfaces before the execution model is proven correct.
- Designing live validator election now:
  rejected because validator admission and consensus are not needed to validate the deterministic state machine.
- Splitting into many crates now:
  rejected because the current feature set is too small to justify the extra boundaries.

## Next Implementation Slice

Implement `flexnet-core` in this order:

1. core types: `Address`, `Account`, `State`, `Transfer`, `Block`
2. canonical byte encoding helpers
3. signature verification for `Transfer`
4. transaction verification and application
5. state hashing
6. block hashing
7. block execution and validation against prior state
8. scenario tests covering valid and invalid flows
