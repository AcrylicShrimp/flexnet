# flexnet-chain state abstraction

## Goal

Reshape `flexnet-chain`'s state boundary so the engine remains generic over storage, but block execution and chain append are actually implementable and testable.

## Problem

The current `StateTransition` trait is too generic:

```rust
pub trait StateTransition {
    fn apply<S>(&self, state: S, delta: StateDelta) -> S
    where
        S: StateView;
}
```

This requires one implementation to accept any `S: StateView` and return that same `S`, but `StateView` only exposes reads. A concrete in-memory state can implement `StateView`, but no honest implementation can mutate or rebuild arbitrary `S`.

## Agreed design

Keep the read abstraction, but replace the transition abstraction.

Use:

```rust
pub trait StateView {
    fn all_accounts_in_order(&self) -> impl Iterator<Item = (Address, Account)>;
    fn get_account(&self, address: &Address) -> Account;
}

pub trait WritableState: StateView {
    fn apply_delta(&mut self, delta: StateDelta);
}
```

Key decisions:

- Keep `StateView` for hashing and verification code that only reads state.
- Replace `StateTransition` with `WritableState`.
- Keep `apply_delta` mutable.
- Remove any `applying` convenience method from the abstraction.
- Make `Chain` mutable on append: `append_block(&mut self, block)`.

## Execution model

`execute_block` should remain atomic from the caller's perspective:

- take the current state as an immutable base reference
- create an in-memory working overlay of pending account updates
- execute each transaction against the merged base-plus-overlay view
- verify the resulting state hash from that merged view
- return the accumulated delta on success

That keeps invalid blocks from partially mutating the canonical chain without requiring full-state cloning.

Recommended shape:

```rust
pub fn execute_block<S>(
    block: &Block,
    config: &ChainConfig,
    previous_state: &S,
) -> Result<ExecutionOutcome, BlockExecuteError>
where
    S: WritableState,
```

Where:

```rust
pub struct ExecutionOutcome {
    pub state_delta: StateDelta,
    pub state_hash: Hash,
}
```

The implementation uses a temporary `WorkingState<'a, S>`:

```rust
pub struct WorkingState<'a, S>
where
    S: StateView,
{
    base: &'a S,
    updates: BTreeMap<Address, Account>,
}
```

`WorkingState` implements:

- `StateView` so transaction execution can read the merged state
- a local mutable `apply_delta` method for accumulating updates during block execution

Read semantics:

- `get_account(address)` returns `updates[address]` if present, otherwise `base.get_account(address)`
- `all_accounts_in_order()` iterates the merged ordered keyspace of `base` and `updates`
- empty accounts are filtered out by hashing code, not hidden from the overlay itself

Commit semantics:

- on success, convert the working overlay into one final `StateDelta`
- the caller applies that delta to canonical state
- on failure, drop the overlay and leave canonical state untouched

## Chain model

`Chain` should keep its current state and mutate only after successful execution.

Recommended shape:

```rust
pub struct Chain<S>
where
    S: WritableState,
{
    config: ChainConfig,
    state: S,
    tip_block: Block,
    tip_block_hash: Hash,
}

pub fn append_block(&mut self, block: Block) -> Result<(), ChainAppendError>
```

Append flow:

1. validate next height
2. validate previous block hash
3. execute block against a temporary overlay working state
4. apply the returned delta to canonical state
5. replace `self.tip_block` and `self.tip_block_hash` only on success

## Why this is simpler

- one state abstraction instead of a read trait plus an impossible transition trait
- no separate transition object flowing through every append call
- mutable canonical updates are explicit
- atomic execution does not require cloning the whole state
- the chain API matches the expected ownership model for a canonical in-memory node
- invalid blocks are easy to reject without rollback machinery

## Rejected alternatives

### Keep `StateTransition` as-is

Rejected because it cannot be implemented honestly for arbitrary `S: StateView`.

### Keep `Chain::append_block(self, ...) -> Result<Self, _>`

Rejected because invalid proposals are normal in consensus simulations. Consuming the chain on each append makes error handling and multi-node simulation unnecessarily awkward.

### Remove state abstraction entirely

Rejected for now. A single mutable state trait still preserves a useful boundary without adding the current impossible abstraction.

### Clone the entire state before every block

Rejected as the primary design because the overlay model preserves atomicity without forcing `Clone` on storage backends or copying the full state on every execution.

## Next implementation slice

1. Delete `StateTransition`.
2. Add `WritableState`.
3. Add one concrete in-memory state implementation inside `flexnet-chain`.
4. Add `WorkingState` as a base-plus-overlay execution view.
5. Change `execute_block` to return `ExecutionOutcome { state_delta, state_hash }`.
6. Change `Chain::append_block` to mutate `&mut self`.
7. Add unit tests for state, transfer, hashing, codec, block execution, and chain append.
8. Add integration tests for multi-block and edge-case scenarios.
