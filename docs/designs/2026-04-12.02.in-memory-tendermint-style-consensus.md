# In-Memory Tendermint-Style Consensus

## Design Goal

Add a simple Tendermint / CometBFT-style consensus layer on top of the existing deterministic execution and chain harness.

The objective is not real networking yet. The objective is:

- multiple validator nodes
- fixed validator set
- deterministic proposer selection
- proposal / prevote / precommit flow
- nil votes
- timeouts
- round increments
- finalized blocks appended into the existing `Chain`

## Scope

This slice is intentionally limited to:

- in-memory message delivery only
- fixed validator set from genesis-like configuration
- equal voting power for all validators
- full-block proposals, not chunked block parts
- logical timeouts driven by a simulation scheduler
- finalized blocks appended into `Chain`

Out of scope for this slice:

- real p2p transport
- cryptographic signatures on consensus messages
- validator set updates
- byzantine behavior injection as a first requirement
- WAL / crash recovery
- block sync / state sync
- Tendermint valid-round optimization

## High-Level Architecture

Keep this inside `flexnet-core` for now.

Add three concrete modules:

- `validator.rs`
- `consensus.rs`
- `network.rs`

### `validator.rs`

Owns:

- `ValidatorId = Address`
- `ValidatorSet`
- quorum calculation
- proposer selection

Required behavior:

- preserve validator order exactly as configured
- proposer for `(height, round)` is deterministic round-robin

Recommended proposer formula:

- `proposer_index = ((height - 1) + round as u128) % validators.len() as u128`

That yields validator `0` as proposer for height `1`, round `0`.

### `consensus.rs`

Owns the validator-local consensus state machine.

Core types:

- `ConsensusStep = Propose | Prevote | Precommit`
- `ConsensusConfig`
- `ConsensusNode`
- `Proposal`
- `Vote`
- `VoteValue = BlockId | Nil`
- `CommitCertificate`

`ConsensusNode` should own:

- its validator id
- the shared fixed `ValidatorSet`
- its local `Chain`
- current `height`
- current `round`
- current `step`
- proposal storage for the current height
- prevote and precommit sets for the current height
- lock state
- timeout configuration
- pending transactions or a concrete block builder input

### `network.rs`

Owns the deterministic in-memory simulation harness.

Core types:

- `Network`
- `Event`
- `Envelope`
- `TimeoutEvent`

Responsibilities:

- deliver in-memory messages between nodes
- schedule and fire logical timeouts
- maintain deterministic event ordering
- run the system until a target height or quiescence

## Core Data Model

### Validator Set

For the first implementation:

- validators are a fixed ordered list of addresses
- all validators have voting power `1`
- quorum is `floor(2n / 3) + 1`

Do not add weighted voting yet.

### Proposal

A proposal is:

- `height`
- `round`
- `proposer`
- full `block`

Important simplification:

- proposal carries the entire block directly
- no separate block-part protocol

### Vote

A vote is:

- `height`
- `round`
- `step`
- `validator`
- `value`

Where:

- `step` is `Prevote` or `Precommit`
- `value` is either `BlockId(Hash)` or `Nil`

### Commit Certificate

Consensus should produce a commit certificate even if the current `Chain` does not store it yet.

A certificate is:

- `height`
- `round`
- `block_id`
- collected precommits that formed quorum

The chain layer can still append only the block for now. The certificate should remain available from the consensus layer for tests and later extensions.

## Node State Machine

Each node runs one height at a time.

Per-height state:

- `height`
- `round`
- `step`
- `locked_round: Option<u32>`
- `locked_block: Option<Block>`
- `locked_block_id: Option<Hash>`
- `proposal_by_round`
- `prevotes_by_round`
- `precommits_by_round`

### Height Start

At the start of height `h`:

- set `round = 0`
- set `step = Propose`
- clear prior height votes and proposals
- clear lock state from the previous height
- schedule propose timeout for `(h, 0)`
- if this node is proposer for `(h, 0)`, build and broadcast a proposal

### Proposal Step

When proposer for `(height, round)`:

- build a block on top of the local chain tip
- broadcast `Proposal`

If there are no pending transactions:

- propose an empty block

Do not propose nil.

On receiving a proposal for the current `(height, round)`:

- validate the block against local chain state by using the existing deterministic execution path
- if valid and either:
  - there is no lock, or
  - the proposal block id equals the locked block id
  then prevote for the block id
- otherwise prevote nil

On propose timeout before sending prevote:

- prevote nil

### Prevote Step

On receiving prevotes for `(height, round)`:

- once quorum prevotes the same block id, if the full proposal block is known and valid:
  - lock that block
  - set `locked_round = round`
  - precommit that block id
- once quorum prevotes nil:
  - precommit nil

On prevote timeout before sending precommit:

- precommit nil

### Precommit Step

On receiving precommits for `(height, round)`:

- once quorum precommits the same block id and the corresponding block is known:
  - commit the block into the local `Chain`
  - emit a `CommitCertificate`
  - advance to `height + 1`

On precommit timeout without commit:

- increment round
- enter `Propose(height, round + 1)`

Locks survive round increments within the same height.

## Nil Vote Semantics

Nil is used only in votes:

- prevote nil
- precommit nil

Nil means:

- no valid proposal seen in time, or
- proposal conflicts with the current lock, or
- no quorum for a block during the step

Nil is never proposed as a block.

## Timeout Model

Use logical deterministic time.

Suggested initial config:

- `propose_timeout_ms = 1000 + round * 250`
- `prevote_timeout_ms = 500 + round * 250`
- `precommit_timeout_ms = 500 + round * 250`

Timeouts should be represented as scheduled events, not sleeps or threads.

Every timeout event must carry:

- `validator`
- `height`
- `round`
- `step`

Stale timeouts must be ignored after a node has moved to a different `(height, round, step)`.

## In-Memory Network Model

The network should be a deterministic event queue.

Event kinds:

- deliver consensus message to a validator
- fire timeout for a validator

For the first implementation:

- broadcast means enqueue one message per validator
- include self-delivery
- use reliable delivery by default
- use deterministic ordering such as `(time, sequence_number, target_index)`

Do not add threads or async runtime.

The entire simulation should run as a pure, single-threaded event loop.

## Integration With Chain

Consensus must not reimplement block execution.

Required integration rule:

- proposal validation uses the existing block validation/execution pipeline
- commit uses `chain.append_block(block)`

Consensus only decides *which* block becomes canonical.
The existing chain and execution layers remain the source of truth for *whether* a block is valid.

## Block Building

Keep block building simple and concrete.

Do not add a generic proposer trait yet.

Use a concrete block builder that:

- takes local pending transfers in existing order
- selects up to `MAX_TRANSACTIONS_PER_BLOCK`
- keeps only transactions valid against the proposer's working state
- produces a valid next block for the current chain tip

This block builder is not consensus. It is only the proposer input source.

## Deliberate Simplifications

These are intentional for the first consensus slice:

1. No consensus message signatures.
   Sender identity is trusted from the in-memory network envelope.
2. No valid-round optimization.
   Proposers do not carry `valid_round` / `valid_value` yet.
3. No block part protocol.
   Proposal includes full block.
4. No weighted voting.
   One validator equals one vote.
5. No byzantine actors required initially.
   Start with honest validators plus timeouts.

These reduce moving parts while preserving the core round logic.

## Key Invariants

- all validators use the same ordered validator set
- proposer selection is deterministic
- quorum threshold is deterministic
- each validator votes at most once per `(height, round, step)`
- a validator may only lock one block per round
- chain mutation happens only on commit
- stale timeouts do not mutate current consensus state
- network event ordering is deterministic

## Recommended Implementation Order

1. `validator.rs`
   - validator set
   - quorum math
   - proposer selection
2. `consensus.rs`
   - message types
   - vote sets
   - node state machine
   - lock handling
3. `network.rs`
   - event queue
   - timeout scheduling
   - reliable broadcast
4. simple block builder
5. integration tests
   - 4 validators, no delay, happy path
   - proposer timeout causing round increment
   - nil prevote / nil precommit path
   - multi-height run appending blocks into `Chain`

## Open Issues

1. Genesis format for validators:
   the current spec has no validator set section yet
2. Commit storage:
   decide later whether commit certificates should be stored alongside blocks
3. Mempool model:
   first proposer input can be a simple local queue, but eventual transaction dissemination is a separate concern

## Rejected Alternatives

- Implementing real networking now:
  rejected because it obscures the consensus state machine behind transport concerns
- Adding weighted validators now:
  rejected because equal voting power is sufficient for the first BFT simulation
- Building a generic consensus abstraction layer:
  rejected because only one algorithm is needed now
- Rewriting the chain layer to store commit proofs immediately:
  rejected because the current chain harness already cleanly accepts finalized blocks
