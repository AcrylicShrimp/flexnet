# Core Crates And Module Layout

## Design Goal

Choose the minimal Rust dependency set and internal crate/module structure for the first Flexnet implementation slice.

This design is constrained by:

- deterministic execution only
- no real networking
- no live consensus yet
- strict alignment with `docs/SPEC.md`
- repository preference for structural simplicity and hard cutovers

## Agreed Workspace Shape

Keep a single implementation crate for now:

- `flexnet-core`

Do not add more crates yet.

Rejected for now:

- `flexnet-crypto`
- `flexnet-types`
- `flexnet-state`
- `flexnet-sim`

These splits would add package boundaries without enough code volume or multiple real implementations to justify them.

## Dependency Set

Use only the crates that map directly to the current spec and testing needs.

### Runtime Dependencies

- `ed25519-dalek = "2"`
  - reason: stable line of the Ed25519 implementation that directly matches the signature scheme in the spec
  - note: `cargo info ed25519-dalek@2` reports `2.2.0`, while the latest published package head is the pre-release `3.0.0-pre.6`
- `sha2 = "0.11"`
  - reason: direct implementation of SHA-256 from RustCrypto
- `serde = "1"`
  - reason: derive support for parsing genesis and fixture data
- `serde_json = "1"`
  - reason: parse genesis JSON and scenario fixtures
- `hex = "0.4"`
  - reason: encode and decode `0x...` address, hash, and signature fields for JSON-facing inputs and debugging
- `thiserror = "2"`
  - reason: explicit structured error enums without handwritten boilerplate

### Dev Dependencies

- `proptest = "1"`
  - reason: property tests for determinism, replay stability, and invalid-input coverage

## Intentionally Not Included

- no async runtime
- no networking crates
- no database or persistence crates
- no Merkle tree crates
- no generic serialization format crates for consensus bytes
- no `rand` dependency in the initial design

Consensus/signature/hashing bytes should be encoded manually from the spec, not delegated to `bincode`, `postcard`, or similar formats.

## Cargo Placement

Put dependency versions directly in `crates/flexnet-core/Cargo.toml` for now.

Reason:

- there is only one real crate
- using `[workspace.dependencies]` now would add indirection without reducing duplication

If a second crate is introduced later and it shares these dependencies, move shared versions up at that point.

## Module Layout

Use a flat, explicit module layout inside `crates/flexnet-core/src/`:

- `lib.rs`
- `constants.rs`
- `error.rs`
- `codec.rs`
- `address.rs`
- `account.rs`
- `transfer.rs`
- `block.rs`
- `genesis.rs`
- `state.rs`
- `hash.rs`
- `signature.rs`
- `execute.rs`

## Module Responsibilities

- `constants.rs`
  - protocol constants from the spec
- `error.rs`
  - all public error enums for transfer, block, genesis, and decoding failures
- `codec.rs`
  - canonical byte encoding helpers for integers and protocol types
  - defines the narrow encoding trait used by canonical byte views
- `address.rs`
  - `Address` type and parsing/formatting helpers
- `account.rs`
  - `Account` type and account-level helpers
- `transfer.rs`
  - `Transfer` data model and transfer-specific verify/apply logic
  - borrowed views for transfer signing bytes and full transaction bytes
- `block.rs`
  - `Block` data model and block-level structural checks
  - borrowed views for block-hash preimage encoding
- `genesis.rs`
  - genesis JSON model, parsing, and conversion into initial chain state
- `state.rs`
  - state container, implicit account behavior, state-delta application, and sorted iteration for hashing
- `hash.rs`
  - `state_hash` and `block_hash`
- `signature.rs`
  - Ed25519 verification and transfer signing payload construction
- `execute.rs`
  - orchestration entry points for block validation and block execution against prior state

## Structural Rules

- keep types concrete
- only introduce traits when multiple concrete implementations are required now
- do not introduce a generic `Transaction` enum while `Transfer` is the only operation
- do not split hashing/signature responsibilities into separate crates
- do not mix JSON serde encoding with canonical consensus encoding

The canonical bytes used for hashing and signatures must remain explicit and hand-authored from the spec.

## Canonical Encoding Model

Adopt a narrow view-based encoding pattern.

### `codec.rs` owns the encoding contract

`codec.rs` should define:

- primitive little-endian writers
- fixed-byte append helpers
- one narrow trait for canonical byte emission, for example `EncodeCanonical`

That trait should only express:

- `encode_into(&self, out: &mut Vec<u8>)`

Do not add a generic pluggable codec framework, format registry, serializer abstraction, or trait hierarchy.

### Structs expose borrowed views only when they have distinct byte domains

Use view types where the same logical value has more than one canonical byte representation.

Examples:

- `TransferSigningView<'a>`
  - encodes `chain_id || chain_version || from || to || amount || nonce`
- `TransferBytesView<'a>`
  - encodes the full transaction bytes including signature
- `BlockHashView<'a>`
  - encodes the block hash preimage using `transactions_hash`

These views should implement the codec trait.

### Do not create views for everything

Avoid mechanical `FooView` wrappers for types that do not need multiple encoding contexts.

Examples of types that should stay plain:

- `Address`
- `Account`
- `State`

`State` hashing is a fold algorithm over sorted entries, not a single canonical serialization of the whole state. Keep that logic in `hash.rs` and `state.rs` instead of forcing it through a generic whole-state view type.

### Why this compromise is the right level

- it preserves explicit protocol bytes
- it keeps encoding rules close to the owning data model
- it avoids duplicating encoding logic across hash and signature code
- it avoids a premature abstraction layer over all domain types
- it satisfies the repo rule because the trait has multiple real implementations immediately

## Test Layout

Use integration tests under `crates/flexnet-core/tests/`:

- `transfer.rs`
- `block.rs`
- `genesis.rs`
- `determinism.rs`

Use these tests to simulate multiple nodes by replaying the same inputs through independent state instances and comparing outputs exactly.

## Tradeoffs

### Why flat modules instead of nested submodules

- fewer moving parts
- easier navigation during the first implementation slice
- no premature taxonomy for a codebase that is still tiny

### Why `Transfer` instead of a generic transaction layer now

- the spec currently defines only one operation
- a single-variant enum or abstract operation layer would be an unused abstraction
- if more transaction kinds are added later, the code can be cut over cleanly then

### Why narrow codec traits are acceptable here

- there are multiple real canonical byte views immediately
- the trait is local, concrete, and single-purpose
- the alternative is scattering nearly identical `to_bytes` logic across unrelated modules
- the trait is for explicit protocol bytes only, not for generic object serialization

### Why JSON support is included now

- the spec already defines genesis in JSON
- scenario tests are easier to read and maintain with JSON fixtures
- JSON is limited to human-facing inputs; it is not used for protocol hashing or signatures
